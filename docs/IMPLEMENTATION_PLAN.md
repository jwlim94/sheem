# SHEEM 구현 계획서

## 1. 제품 개요

**SHEEM**은 하루를 차분하게 마무리할 수 있도록 돕는 리플렉션 웹 앱입니다.

### 핵심 컨셉

- 사용자가 짧은 텍스트 또는 음성으로 하루를 돌아보면
- AI가 감정을 분석하여 **추상적인 이미지 한 장**으로 변환
- 이 이미지가 그날의 기록이 됨

### 핵심 가치

- "하루를 놓치지 않았다"
- "하루를 조용히 마무리했다"
- 글을 잘 써야 한다는 부담 없음
- 비교나 퍼포먼스 압박 없음

### 브랜드 톤

- 이름: **Sheem** (한국어 "쉼"에서 유래)
- 키워드: 여운, 온기, 여백, 잔향, 차분함

## 3. 매일 성찰 질문

### 개념

- 하루 1개의 부드러운 성찰 질문 제공
- 답변은 **선택적** (강제 아님)
- 질문에 답하면 일기 작성의 진입 장벽 낮춤

### 질문 예시

| 유형 | 예시 |
|------|------|
| 감정 | "오늘 마음에 남은 순간이 있나요?" |
| 감사 | "오늘 작은 감사함을 느낀 적이 있나요?" |
| 관계 | "오늘 누군가와 나눈 대화가 떠오르나요?" |
| 성찰 | "오늘의 나에게 해주고 싶은 말이 있나요?" |

### 규칙

- 질문은 매일 바뀜 (시스템에서 제공)
- 답변 여부는 기록에 영향 없음
- 질문 + 일기 모두 하나의 이미지로 합쳐짐

## 4. 부드러운 성장 시스템

### 철학

- **압박 없는 동기 부여**
- 기록하면 보상, 안 해도 벌칙 없음
- 스트릭/포인트 감소 없음

### "조용한 정원" 컨셉

```
┌─────────────────────────────────────┐
│     🌸  🌷     🌼                   │
│        🌻   🌺                      │
│   🌹        🪻    🌸                │
│                                     │
│  "1월에 12송이의 꽃이 피었어요"      │
└─────────────────────────────────────┘
```

| 요소 | 설명 |
|------|------|
| 꽃 추가 | 기록할 때마다 정원에 꽃 1송이 추가 |
| 꽃 유지 | 안 해도 기존 꽃은 **그대로 유지** (시들지 않음) |
| 계절 흔적 | 봄/여름/가을/겨울 각 계절에 기록하면 자연스럽게 모임 |
| 시각적 성장 | 정원이 점점 풍성해지는 느낌 |

### 의도적으로 제외

| 요소 | 이유 |
|------|------|
| 스트릭 카운터 | "놓치면 잃는다" 압박 유발 |
| 포인트 감소 | 벌칙 느낌 → 죄책감 |
| 랭킹/비교 | 타인과 비교 → SHEEM 철학과 충돌 |

## 5. 템플릿 시스템

### 개념

- 템플릿 = 시각적 "문법" (일관된 스타일)
- 내용 = 감정에 따른 "단어" (다이나믹하게 변화)
- 같은 템플릿이라도 감정에 따라 색상, 구도, 강도가 달라짐

### 초기 템플릿 (예시)

| 이름 | 스타일 | 설명 |
|------|--------|------|
| **Afterglow** | 소프트 그라디언트 | 따뜻한 색감, 빛 번짐 효과 |
| **Ink** | 수묵화/미니멀 | 여백 중심, 단색 계열 |
| **Fragment** | 타이포그래피 | 감정 키워드를 시각적으로 배치 |
| **Dream** | 몽환적/일러스트 | 애니메이션풍, 부드러운 선 |
| **Raw** | 텍스처/질감 | 캔버스 느낌, 물감 터치 |

### 선택 시점

- 매일 변경 가능
- 기본 템플릿 설정 가능

## 8. 데이터 모델 (초안)

### users

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | uuid | PK |
| email | string | 이메일 |
| created_at | timestamp | 가입일 |
| default_template | string | 기본 템플릿 |

### entries

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | uuid | PK |
| user_id | uuid | FK → users |
| date | date | 날짜 (하루 1개 제한) |
| question_id | uuid | FK → questions (nullable) |
| question_response | text | 질문에 대한 답변 (nullable) |
| input_text | text | 일기 원본 텍스트 |
| input_type | enum | 'text' / 'voice' |
| emotion_data | jsonb | 감정 분석 결과 |
| template | string | 사용한 템플릿 |
| image_url | string | 생성된 이미지 URL |
| created_at | timestamp | 생성 시간 |

### questions

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | uuid | PK |
| content | text | 질문 내용 |
| category | string | 질문 유형 (감정/감사/관계/성찰) |
| is_active | boolean | 활성화 여부 |

### templates

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | string | PK (예: 'afterglow') |
| name | string | 표시 이름 |
| description | text | 설명 |
| style_prompt | text | SDXL 스타일 프롬프트 |
| is_active | boolean | 활성화 여부 |

### garden_flowers

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | uuid | PK |
| user_id | uuid | FK → users |
| entry_id | uuid | FK → entries |
| flower_type | string | 꽃 종류 |
| season | string | 계절 (spring/summer/fall/winter) |
| position_x | float | 정원 내 X 위치 |
| position_y | float | 정원 내 Y 위치 |
| created_at | timestamp | 생성 시간 |

## 9. API 설계 (초안)

### Next.js → FastAPI

#### POST /api/generate

하루 기록 생성 요청

**Request:**

```json
{
  "user_id": "uuid",
  "question_id": "uuid (optional)",
  "question_response": "질문 답변 (optional)",
  "input_type": "text" | "voice",
  "input_data": "텍스트 또는 음성 파일 URL",
  "template": "afterglow",
  "date": "2025-01-21"
}
```

**Response:**

```json
{
  "success": true,
  "entry": {
    "id": "uuid",
    "image_url": "https://r2.example.com/...",
    "emotion_data": {
      "primary_emotion": "nostalgia",
      "energy": "low-medium",
      "warmth": "warm",
      "keywords": ["connection", "parting"]
    },
    "flower": {
      "type": "tulip",
      "season": "winter"
    }
  }
}
```

## 12. 핵심 원칙

1. **단순함**: 하루 1회, 입력 1개, 이미지 1장
2. **부담 없음**: 글을 잘 쓸 필요 없음, 길게 쓸 필요 없음
3. **추상화**: 직접적인 기록이 아닌 감정의 시각화
4. **프라이버시**: 기본은 개인 기록, 공유는 선택
5. **비교 없음**: 좋아요 수 숨김, 스토리텔링 경쟁 방지
6. **부드러운 성장**: 압박 없이 쌓이는 느낌
